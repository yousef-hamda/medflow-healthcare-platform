# medflow-ml (ml-batch)

Batch ML for MedFlow: feature engineering, model training, cohort building and
drift monitoring. Runs as a set of one-shot jobs that read the OMOP **gold**
Delta lakehouse, train models, and register them in MLflow for `ml-serving`.

> **All data is synthetic.** Tabular/clinical data is generated by
> [Synthea](https://github.com/synthetichealth/synthea) (seed 42); imaging uses
> the **NIH ChestX-ray14** dataset, which is **research-use-only** and never
> redistributed here. Nothing in this package is cleared for clinical use.

## Design

The package is built around **shared, dependency-free feature functions** in
`medflow_ml.features.*`. These are the *training-time source of truth* and are
mirrored byte-for-byte at serving time by `medflow_serving.inference.featurize`.
Constants such as `WINDOW_HOURS`, `RESAMPLE_MINUTES`, `SEQUENCE_STEPS`,
`VITALS_FEATURES`, `LAB_FEATURES`, `POPULATION_NORMALS` and the z-score scales
MUST stay identical across both packages, otherwise online scores diverge from
offline training. The unit tests (`tests/`) pin these invariants and assert that
every windowed feature uses only **pre-prediction-time** data (no label
leakage).

```
src/medflow_ml/
  config.py              pydantic-settings (MLflow, MinIO, Spark, Feast, ChestX-ray dir)
  data_io/               Spark + Delta readers/writers (s3a:// against MinIO)
  features/
    vitals.py            resample/normalize/rolling-stats/slope (shared with serving)
    labs.py              lab abnormality flags vs adult reference ranges
    encounters.py        age bands, comorbidity flags, prior-admission counts
  models/lstm.py         PyTorch-Lightning sepsis LSTM (2x64, dropout 0.3)
  evaluation/metrics.py  AUROC/AUPRC, calibration, ECE, subgroup AUROC (numpy only)
  jobs/                  the runnable entrypoints (below)
```

## Jobs

Each job is a module run with `python -m medflow_ml.jobs.<name>`. In the dev
stack they run inside the `ml-batch` container via `docker compose run --rm`.

| Job | Module | What it does |
| --- | --- | --- |
| **Build cohorts** | `medflow_ml.jobs.build_cohorts` | Materializes reusable OMOP cohorts (`sepsis_admissions`, `readmission_index`, `adult_inpatient`) to `gold/cohorts/<name>` as partitioned Delta. |
| **Backfill features** | `medflow_ml.jobs.backfill_features` | Computes rolling vitals stats (1h/6h/24h mean/min/max/slope), prior-encounter counts and lab flags; writes parquet to the offline store and runs `feast materialize` into Redis. All windows are strictly historical (point-in-time correct). |
| **Train sepsis** | `medflow_ml.jobs.train_sepsis` | Resamples 6h vitals windows (24×5) and trains the sepsis-EWS LSTM with a time-aware split; logs AUROC/AUPRC to MLflow and registers `sepsis-ews`. |
| **Train readmission** | `medflow_ml.jobs.train_readmission` | Engineers encounter features (strictly historical look-backs), trains a calibrated XGBoost model, logs SHAP + subgroup metrics and registers `readmission-30d`. |
| **Train X-ray** | `medflow_ml.jobs.train_xray` | Fine-tunes DenseNet121 on a local **ChestX-ray14** mirror (`CHESTXRAY_DIR`) for 14 findings, patient-disjoint split; registers `chest-xray-14`. |
| **Drift report** | `medflow_ml.jobs.drift_report` | Compares recent serving inputs (predictions log) against the training baseline with Evidently and uploads HTML to `s3://drift-reports/{date}/{model}.html`. |

## Running via `make`

From the repo root (these use `docker compose run --rm ml-batch ...`):

```bash
make spark              # bring up the Spark master + worker first
make train-sepsis       # python -m medflow_ml.jobs.train_sepsis
make train-readmission  # python -m medflow_ml.jobs.train_readmission
make download-chestxray # fetch a small research-use-only ChestX-ray14 slice
make train-xray         # python -m medflow_ml.jobs.train_xray
```

Cohort, backfill and drift jobs are run directly:

```bash
docker compose run --rm ml-batch python -m medflow_ml.jobs.build_cohorts
docker compose run --rm ml-batch python -m medflow_ml.jobs.backfill_features
docker compose run --rm ml-batch python -m medflow_ml.jobs.drift_report --model sepsis-ews
```

## Configuration

Copy `.env.example` to `.env` and adjust. Key variables: `MLFLOW_TRACKING_URI`,
`MLFLOW_S3_ENDPOINT_URL`, `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`,
`SPARK_MASTER`, `DELTA_S3_ENDPOINT`, `CHESTXRAY_DIR`. See `config.py` for the
full set and defaults.

## Container

The image is multi-stage `python:3.11-slim`. The runtime stage also installs a
headless JDK 17 because the PySpark jobs run as a **Spark client/driver** that
submits to the external `spark-master`; `JAVA_HOME` is set for the Spark
launcher. Torch CPU wheels are used to keep the image small.

## Tests

```bash
cd apps/ml-batch
pytest                  # windowing/slope, leakage guard, evaluation math, age bands
```

`tests/conftest.py` provides small synthetic pandas frames (vitals, encounters,
labels) shared across tests. No network, Spark or MLflow is required for the
unit tests — they exercise the pure feature/evaluation functions only.
